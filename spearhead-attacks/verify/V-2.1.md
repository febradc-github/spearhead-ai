## Root cause

This is a real, direct consequence of T-2's own change, not an unrelated flake.
`mcp-server/server.js`'s `runSearch` calls `rankBySimilarity(index, queryEmbedding,
effectiveLimit)` **without** a `minScore` argument (T-3's job is to pass an explicit,
env-var-resolved one). Per DESIGN.md's explicit, approved decision ("Does
`rankBySimilarity` get a default `minScore` even when called without one? Yes —
defaulting to `DEFAULT_MIN_SCORE` makes the function safe-by-default"), an
omitted `minScore` now defaults to `0.5` rather than the old no-cutoff behavior.

`server.test.js`'s existing fixtures deliberately use simple, easy-to-hand-verify
vectors (`[1,0,0]` vs `[0,1,0]`, `[1,0]` vs `[1,i]` for `i` up to 11) — several of
these pairs are orthogonal or near-orthogonal, scoring at or near `0`, which is
below the new `0.5` default. Those fixtures were correct against the old
no-cutoff behavior and are now correctly excluded by the new cutoff; the tests'
*expectations* (`results.length === 2`, `=== 8`) are what's stale, not the new
production code's behavior.

The coder flagged this exact effect in their completion report as informational
(not a scope violation, since `server.js`/`server.test.js` are T-2's explicit
out-of-scope files) — this verify pass confirms it manifests as a genuine
full-suite mechanical-gate failure when T-2 is tested/merged in isolation.
