## Goal

Write `agents/guru.md`, a new sub-agent that searches the knowledge base, cross-checks staleness, falls back to source, and documents findings.

## Expected files

- `agents/guru.md`

## Depends on

none

## Acceptance criteria

- Follows the same frontmatter/structure pattern as `agents/spearhead-scout.md` (`name`, `description`, `model`, `effort` frontmatter; isolation-justification line; numbered constraints; process/reporting sections).
- `description` states `guru` is dispatched via the Agent tool by an already-running agent session, and explicitly says "never invoke directly" / has no direct user-facing command (matching `spearhead-scout`/`-coder`/`-verifier`'s convention).
- Documented process: (1) search `spearhead-knowledge/**/*.md` via `Glob`/`Grep`/`Read` for notes relevant to the given query; (2) for any match, cross-check its `source_hash` frontmatter against a freshly-computed hash (via `lib/hash.js`'s `hashContent`) of the file at its `source:` path — a mismatch, or a `source:` path that no longer exists, means the note is stale; (3) if nothing relevant is found, or the only matches are stale, fall back to reading/grepping the actual source/repo to find the answer; (4) on a successful fallback, write a new `code/` note (if none existed) or refresh the existing stale one in place plus a `## Changelog` entry, using `scripts/knowledge-path.js` for deterministic naming; (5) return an answer grounded in whichever path was taken (fresh knowledge, refreshed knowledge, or freshly-discovered source).
- Explicitly states `guru` only writes/refreshes `code/` notes — `decisions/`/`design/`/`architecture/` notes are out of scope for it, staying agent-judgment as they've been since A-1.
- Explicitly states `guru` never fabricates or writes a note it isn't confident about — if the source-fallback search also finds nothing, it reports that plainly rather than writing something speculative.
- Documents the kimi-code fallback: since kimi-code doesn't support plugin-defined sub-agents, the process above is written clearly enough that a calling agent under kimi-code can perform the same steps inline, in its own session, without a dispatch mechanism.
- References `lib/hash.js` (the relocated location, not `mcp-server/lib/hash.js`) for hashing.

## Out of scope

- Any code file — this task is a single markdown agent-definition file.
- Actually relocating `lib/hash.js` (T-2's job) — this task's prose references the target location, doesn't perform the move.
- Any change to `hooks/knowledge-nudge.js`, `hooks/remind.js`, `scripts/knowledge-path.js`, or `lib/knowledge-frontmatter.js` — referenced, not modified, by this task.

## Verification commands

- Manual structural review: confirm `agents/guru.md` exists with valid YAML frontmatter (`name: guru`, `description`, `model`, `effort`) matching the pattern in `agents/spearhead-scout.md`, `agents/spearhead-coder.md`, `agents/spearhead-verifier.md`. Agent definitions in this repo are prompt specs, not unit-tested code — no `.test.js` file is expected or required for this task.
