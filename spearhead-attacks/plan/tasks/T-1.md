## Goal

Add a `source_hash` scalar frontmatter field to `lib/knowledge-frontmatter.js`, parsed and serialized the same way `source`/`updated` already are.

## Expected files

- `lib/knowledge-frontmatter.js`
- `lib/knowledge-frontmatter.test.js`

## Depends on

none

## Acceptance criteria

- `parseFrontmatter` recognizes `source_hash` as a scalar field (added to the existing `SCALAR_FIELDS` set), parsed the same way as `source`/`updated`.
- `serializeFrontmatter` writes `source_hash` the same way it writes `source`/`updated` (present only when provided).
- A note with `source_hash` round-trips through `parse(serialize(x))` reproducing the field exactly, same as the existing fields.
- A note with no `source_hash` field parses without error and yields the field as `undefined` (not a thrown error, not a default value) — matching the existing "optional fields present only when provided" behavior.
- Malformed frontmatter still falls back to `{type: 'unknown'}` as it does today; `source_hash` is not treated specially in the fallback path.

## Out of scope

- Anything in `hooks/knowledge-nudge.js` (T-2 in this attack — that task computes and compares the hash; this task only stores/retrieves the field).
- `mcp-server/lib/hash.js` (unchanged, only consumed elsewhere).

## Verification commands

- `node lib/knowledge-frontmatter.test.js`
