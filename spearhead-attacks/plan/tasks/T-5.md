## Goal

Wire hashing, embeddings, and the index store together into a file-watch
pipeline that keeps `embeddings.json` in sync with the three knowledge
sources, hash-gated and queued sequentially.

## Expected files

- `mcp-server/lib/watch.js`
- `mcp-server/lib/watch.test.js`
- `mcp-server/lib/pipeline.js`
- `mcp-server/lib/pipeline.test.js`

## Depends on

T-2, T-3, T-4

## Acceptance criteria

- `watch.js` recursively watches (`fs.watch`) `spearhead-knowledge/**/*.md`,
  `spearhead-attacks/**/*.md`, and configured general-doc paths (`README.md`,
  `docs/**/*.md`).
- `pipeline.js`, on a create/change event: computes the content hash; if it
  matches the existing index entry, skips (no embeddings call); otherwise
  calls the embeddings module and updates the index store (PROBLEM.md
  acceptance criteria 1, 3).
- Events are processed sequentially via an in-memory queue, not fired as
  concurrent embeddings calls (DESIGN.md failure-mode: load spike).
- On an embeddings-call failure, the entry is marked `pending` (not treated
  as up to date) so it retries on the next relevant event, and the failure
  does not crash the watcher (DESIGN.md failure-mode handling).
- On startup, re-scans watched paths and reconciles against the index
  (self-heals after a crash — same hash-comparison path as normal
  incremental updates).
- Tests use a temp directory and an injected/mocked embeddings client; no
  live network calls.

- Frontmatter authoring by agents (hooks, T-8/T-9).
