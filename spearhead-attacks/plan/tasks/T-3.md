## Goal

Wire `mcp-server/server.js`'s `runSearch` to resolve a threshold from `SPEARHEAD_SEARCH_MIN_SCORE` (via T-2's `rankBySimilarity`/`DEFAULT_MIN_SCORE`) and update `SEARCH_TOOL.description` to document the new empty/short-result contract.

## Expected files

- `mcp-server/server.js`
- `mcp-server/server.test.js`

## Depends on

T-2

## Acceptance criteria

- `runSearch` resolves the effective threshold from `process.env.SPEARHEAD_SEARCH_MIN_SCORE`, parsed as a float via `parseFloat`, falling back to `DEFAULT_MIN_SCORE` (imported from `mcp-server/lib/similarity.js`, not re-declared) when unset or `NaN` — never throwing and never disabling the cutoff (PROBLEM.md criterion 4).
- `runSearch` passes the resolved threshold into `rankBySimilarity` as its `minScore` argument.
- `SEARCH_TOOL.description` documents that results are filtered by a minimum relevance score and that an empty result means no sufficiently relevant match was found, not a tool malfunction (PROBLEM.md criterion 5).
- No change to the existing error contract: `MissingApiKeyError`/`EmbeddingsRequestError` still surface as named `isError: true` tool errors exactly as today; an empty-due-to-threshold result is a distinct, successful (non-error) response with `results: []`.
- Tests cover: below-threshold entries excluded from a live `runSearch`/tool call, at/above-threshold entries included, an all-below-threshold query returning `results: []` as a successful (non-error) response, threshold override via `SPEARHEAD_SEARCH_MIN_SCORE` changing which entries are included, and an unset/unparseable env var falling back to the default threshold.

## Out of scope

- Any change to `mcp-server/lib/similarity.js` itself beyond what T-2 already shipped (this task only consumes T-2's new signature/export).
- **(Amendment note)** T-2 (amended after V-2.1) already repaired two pre-existing `server.test.js` tests whose fixtures broke under the new default cutoff (`search tool ranks fixture index entries...`, `search tool defaults to the top 8 results...`). This task should not need to touch those two tests again — it adds the new tests named in this task's acceptance criteria (env var override, unset/unparseable fallback, at/above/below-threshold via a live tool call) alongside them.
- Any change to embedding, indexing, or storage logic.
- Enforcing search-before-read or gating `Read` on empty results (`hooks/guard.js`, `hooks/knowledge-nudge.js` — untouched, separate future attack).
- A per-call `minScore` argument on the `search` tool call itself.

## Verification commands

- `node mcp-server/server.test.js`
