## Goal

Add `/spearhead:obsidian-graph`: a command+skill pair backed by a dependency-free script that opens Obsidian directly to the graph view.

## Expected files

- `scripts/obsidian-graph.js` (create)
- `scripts/obsidian-graph.test.js` (create)
- `commands/obsidian-graph.md` (create)
- `skills/spearhead-obsidian-graph/SKILL.md` (create)

## Depends on

none

## Acceptance criteria

- `scripts/obsidian-graph.js` is a dependency-free Node script (no npm packages) that: detects `process.platform` (`darwin` → `open`, `linux` → `xdg-open`, `win32` → `start`); constructs an `obsidian://advanced-uri?vault=<repo-root-dir-name>&commandid=graph:open` URI, where `<repo-root-dir-name>` is the basename of the resolved project root (same root-resolution convention as `scripts/state.js`/`scripts/knowledge-path.js`); invokes the platform's URI-open command via `node:child_process` with an injectable seam (`options.exec`, mirroring the pattern established in `mcp-server/lib/rank.js`'s `options.exec`, so tests never actually spawn a process or open Obsidian).
- On an unsupported/undetected platform, the script does not crash — it prints the constructed `obsidian://` URI and a message instructing the user to open it manually.
- The script's own output/return value clearly states the two documented preconditions (Obsidian installed; the Advanced URI community plugin installed and enabled) and that it cannot verify Obsidian actually reached the graph view (URI dispatch is fire-and-forget at the OS level) — it only confirms the platform-open command itself was invoked without a spawn error.
- `commands/obsidian-graph.md` + `skills/spearhead-obsidian-graph/SKILL.md` follow this repo's existing command/skill pairing convention (thin command wrapper dispatching to the skill via the Skill tool; skill frontmatter includes `user-invocable: false`); the skill's process runs `scripts/obsidian-graph.js` and relays its output, including the precondition/limitation messaging.
- Tests cover: correct `obsidian://` URI construction (vault name = repo root directory's basename, `commandid=graph:open`); correct platform-to-open-command mapping for `darwin`/`linux`/`win32` via the injected `options.exec` seam (no real subprocess spawn); the unsupported-platform fallback path (prints URI, doesn't throw); a spawn-error case surfacing clearly rather than silently succeeding.

## Out of scope

- Verifying the exact Advanced URI `commandid` value (`graph:open`) against Obsidian's own live command list or Advanced URI's official docs beyond what DESIGN.md already flagged — if the coder implementing this task can verify it directly (e.g., checking installed Obsidian/plugin documentation, if available in this environment), do so and note the finding; otherwise implement as designed and record the assumption plainly rather than silently guessing something different.
- Installing Obsidian or the Advanced URI plugin — user-side preconditions, documented not automated.
- Any change to `agents/guru.md`, `hooks/remind.js`, `rules/RULES.md`, `lib/knowledge-frontmatter.js`, or `mcp-server/` — unrelated to this task.
