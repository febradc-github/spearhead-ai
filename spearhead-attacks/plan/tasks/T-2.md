## Goal

Add a minimum-score cutoff to `rankBySimilarity` in `mcp-server/lib/similarity.js`, excluding entries below `minScore` before sorting/truncating to `limit`.

## Expected files

- `mcp-server/lib/similarity.js`
- `mcp-server/lib/similarity.test.js`
- `mcp-server/server.test.js` (fixture-expectation fix only â€” see amendment note below)

## Depends on

none

## Acceptance criteria

- `similarity.js` exports `DEFAULT_MIN_SCORE = 0.5` alongside the existing `DEFAULT_LIMIT`.
- `rankBySimilarity(index, queryEmbedding, limit = DEFAULT_LIMIT, minScore = DEFAULT_MIN_SCORE)` excludes any entry whose cosine similarity is `< minScore` from its returned array, regardless of `limit` (PROBLEM.md criterion 1).
- Entries at or above `minScore` are still returned, sorted highest-score first, truncated to `limit` â€” unchanged behavior for entries that pass the cutoff (PROBLEM.md criterion 2).
- The cutoff is applied before the `limit` truncation, so fewer than `limit` results (including zero) is possible and expected when few/no entries clear the threshold (PROBLEM.md criterion 4's ordering requirement, at the unit level).
- A query where every indexed entry scores below `minScore` returns an empty array, not an error and not a non-empty array of weak matches (PROBLEM.md criterion 3).
- A `NaN` or otherwise non-numeric score is treated as excluded (fails the `>= minScore` comparison), never thrown or included.
- Existing behavior for calls that omit `minScore` still returns a cutoff-filtered result (using the new default), not the old no-cutoff behavior â€” this is an intentional, documented behavior change; existing tests that assumed no cutoff are updated to pass an explicit low `minScore` (e.g. `-1`) where they need to observe uncut ranking/limit behavior in isolation.
- **(Amendment, added after V-2.1's mechanical gate failure)** `mcp-server/server.js`'s `runSearch` calls `rankBySimilarity` without an explicit `minScore`, so it now inherits the new `DEFAULT_MIN_SCORE = 0.5` default (an intended, approved consequence of this task's design, not a bug). Two pre-existing tests in `mcp-server/server.test.js` used fixture vectors that are orthogonal or near-orthogonal to their query (scoring at/near `0`), which the old no-cutoff behavior returned but the new default correctly excludes: `search tool ranks fixture index entries by similarity and returns {path, excerpt, score}` and `search tool defaults to the top 8 results and honors a custom limit`. This task updates ONLY those two tests' fixtures/expectations (e.g. giving `far.md` an embedding that scores `>= 0.5` against the query if it's meant to still appear, or asserting the now-correct shorter/reordered result set) so the full suite passes. No other part of `server.js` or `server.test.js` changes â€” that remains T-3's scope (env var wiring, tool description text).
