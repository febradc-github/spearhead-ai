## Goal

A new hook, `hooks/knowledge-nudge.js`, that nudges code documentation on a
source file's first read and nudges a doc/changelog update when a task
transitions to `done`.

## Expected files

- `hooks/knowledge-nudge.js`
- `hooks/knowledge-nudge.test.js`
- `.claude-plugin/plugin.json`
- `.kimi-plugin/plugin.json`

## Depends on

T-1, T-7

## Acceptance criteria

- Registered as `PostToolUse` in both plugin manifests with two matchers:
  `Read` and `Bash|PowerShell`.
- On a `Read` of a source file (extension heuristic excluding `.md` and
  config/lockfiles) with no existing note under `spearhead-knowledge/code/`
  whose `source:` matches it, injects a nudge naming the exact target path
  (computed via `scripts/knowledge-path.js`) (PROBLEM.md acceptance
  criterion 4).
- Re-reads of an already-documented, unchanged file in the same session do
  not re-nudge (session-scoped tracking, same idle-expiry pattern as
  `remind.js`) (PROBLEM.md acceptance criterion 6).
- On a `Bash`/`PowerShell` call matching a successful
  `state.js transition <T-id> done` invocation, reads the task's expected
  files from `spearhead/status.yml` (read-only) and nudges the agent to
  update each touched file's code doc with a new `## Changelog` entry
  (PROBLEM.md acceptance criterion 12).
- Never writes `spearhead/status.yml` and never calls the embeddings API
  directly — nudge only (DESIGN.md ADR-003); completes within the existing
  10s hook timeout.
- Tests use `spawnSync` against fixture PostToolUse payloads for both
  matchers (same pattern as existing hook tests per repo convention).

## Out of scope

- The MCP server or index/search logic (T-1, T-3–T-6).
- The search-first reminder (T-8, already covers that via `remind.js`).
